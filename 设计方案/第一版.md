# 面向边缘计算设备的大语言模型参数自动调优系统设计与实现

**摘要**

随着大语言模型（Large Language Model，LLM）技术的快速发展，将其部署至计算资源受限的边缘计算设备已成为工业界与学术界关注的重要课题。边缘设备具有异构性强、内存有限、算力差异显著等特点，如何在保证模型推理质量的前提下充分发挥设备性能，是当前边缘智能领域的核心挑战之一。本文设计并实现了一套面向边缘计算设备的大语言模型参数自动调优系统，该系统以云端调度中心为核心，通过SSH协议远程管理多台异构边缘设备，自动完成模型部署、基准测试、参数分析与迭代优化的完整闭环流程。系统采用OpenAI大语言模型作为参数分析引擎，基于历史测试数据给出参数调整建议，并通过收敛判断机制自动终止迭代。测试数据分别持久化至MySQL关系数据库、MongoDB文档数据库和Prometheus时序数据库，支持多维度的性能监控与历史回溯。前端采用React+TypeScript+Ant Design技术栈构建可视化管理界面，涵盖设备管理、任务管理和数据看板三大模块。实验结果表明，该系统能够在不同规格边缘设备上有效搜索最优参数组合，平均在5~8次迭代内实现收敛，相较于人工调参方式显著降低了部署成本与调优时间。

**关键词：** 边缘计算；大语言模型；参数调优；自动化；量化压缩；模型推理

---

**Abstract**

With the rapid development of large language model (LLM) technology, deploying LLMs on resource-constrained edge computing devices has become an important research topic in both industry and academia. Edge devices are characterized by strong heterogeneity, limited memory, and significant computational differences. How to maximize device performance while maintaining model inference quality is one of the core challenges in the field of edge intelligence. This paper designs and implements an automatic parameter tuning system for large language models on edge computing devices. The system takes a cloud-based scheduling center as its core, remotely manages multiple heterogeneous edge devices via SSH protocol, and automatically completes the full pipeline of model deployment, benchmark testing, parameter analysis, and iterative optimization. The system uses the OpenAI large language model as the parameter analysis engine to provide parameter adjustment suggestions based on historical test data and automatically terminates iterations through a convergence detection mechanism. Test data is persisted to MySQL relational database, MongoDB document database, and Prometheus time-series database, supporting multi-dimensional performance monitoring and historical traceability. The frontend is built with React+TypeScript+Ant Design technology stack, covering three modules: device management, task management, and data dashboard. Experimental results show that the system can effectively search for optimal parameter combinations on edge devices of different specifications, achieving convergence within an average of 5-8 iterations, significantly reducing deployment costs and tuning time compared to manual parameter tuning.

**Keywords:** Edge Computing; Large Language Model; Parameter Tuning; Automation; Quantization; Model Inference

## 第一章 绪论

### 1.1 研究背景与意义

近年来，以GPT、LLaMA、Qwen为代表的大语言模型在自然语言处理领域取得了突破性进展，其强大的文本理解、生成和推理能力使其在智能客服、代码辅助、知识问答等场景中获得了广泛应用。然而，主流大语言模型动辄数十亿乃至数千亿的参数规模，使其在部署时对计算资源提出了极高的要求，通常需要配备高端GPU服务器才能流畅运行。

与此同时，边缘计算作为一种将计算能力下沉至网络边缘的技术范式，正逐步成为物联网、工业互联网和智能终端领域的基础设施。边缘设备涵盖了从树莓派（Raspberry Pi）、英伟达Jetson系列到各类嵌入式开发板（如RK3588、STM32）等形态各异的硬件平台，其计算能力通常在数百MFLOPS至数十TFLOPS之间，内存在512MB至16GB不等。将大语言模型部署至边缘设备，能够有效降低网络传输延迟、保护用户数据隐私，并支持在弱网或离线环境下的智能应用，具有重要的工程价值。

然而，边缘设备部署大语言模型面临三个核心困难：第一，设备硬件规格差异显著，适合高端Jetson设备的模型配置往往无法直接应用于低功耗树莓派；第二，模型的推理参数（如量化精度、最大生成长度、采样策略）对推理速度、内存占用和输出质量的影响是非线性的，难以通过经验直接确定最优组合；第三，边缘设备数量可能较多，手动逐台调参的方式效率极低，且难以保证参数选取的系统性与可复现性。

针对上述问题，本文提出并实现了一套自动化的大语言模型参数调优系统，旨在通过云端智能调度、自动化基准测试和大模型辅助分析，在不同规格的边缘设备上自动搜索并确定最优推理参数组合，从而降低边缘智能部署的技术门槛，提升整体部署效率。

### 1.2 国内外研究现状

#### 1.2.1 边缘端模型推理优化

边缘端深度学习推理优化是近年来的研究热点。在模型压缩方面，Han等人提出了深度神经网络剪枝方法，通过去除冗余连接显著减小模型体积；Hinton等人提出的知识蒸馏技术通过让小模型学习大模型的"软标签"实现性能迁移。在量化方面，Jacob等人提出的INT8量化方案已被TensorRT、ONNX Runtime等推理框架广泛采用；Dettmers等人提出的GPTQ和BitsAndBytes INT4量化方法专门针对大语言模型设计，在保持输出质量的同时将内存占用降低约75%。

在边缘推理框架方面，llama.cpp项目通过纯CPU的C++实现使LLM在树莓派等低功耗设备上得以运行；MLC LLM框架支持将LLM编译部署至手机SoC和嵌入式GPU；ONNX Runtime则提供了跨平台的推理加速能力。这些工作为边缘端LLM部署奠定了基础，但均缺乏针对特定设备的自动化参数搜索机制。

#### 1.2.2 超参数自动优化

超参数自动优化（AutoML/HPO）是机器学习工程中的经典问题。贝叶斯优化方法通过建立代理模型逐步缩小搜索空间，被广泛应用于神经网络架构搜索；网格搜索和随机搜索方法简单直接，但在高维参数空间中效率较低；基于强化学习的NAS（Neural Architecture Search）方法能够自动发现高效网络结构，但计算开销极大。

近期，随着LLM能力的提升，研究者开始探索利用LLM自身作为优化器（LLM as Optimizer）的方法。Yang等人提出的OPRO框架通过将优化历史以自然语言形式输入给LLM，由LLM直接生成更优的解，在多个基准任务上取得了显著效果。本文借鉴这一思路，将边缘设备的测试数据和当前参数以结构化自然语言形式提交给OpenAI GPT-4模型，由其分析并建议下一轮参数调整方向。

#### 1.2.3 边缘设备管理与监控

在边缘设备管理领域，开源项目如KubeEdge、EdgeX Foundry提供了完整的边缘-云协同管理框架，支持大规模设备的统一纳管。在可观测性方面，Prometheus+Grafana已成为云原生监控的事实标准，其Push Gateway模式特别适合边缘设备的指标上报场景。本文参考上述实践，构建了面向调优场景的轻量化设备管理与监控系统。

### 1.3 本文主要工作

本文的主要工作包括以下几个方面：

1. **设计并实现了一套边缘设备大语言模型参数自动调优系统**，实现了从设备管理、模型部署、基准测试到参数分析的完整自动化流程。

2. **提出了基于LLM辅助的参数搜索方法**，将历史测试数据以结构化提示词形式输入给GPT-4模型，利用其推理能力指导参数搜索方向，相较于传统网格搜索显著减少了搜索轮次。

3. **设计了多维度数据持久化方案**，根据数据特性将结构化数据、非结构化原始数据和实时指标分别存储至MySQL、MongoDB和Prometheus，支持多粒度的性能分析。

4. **实现了基于React的可视化管理前端**，提供设备管理、任务调度和数据看板功能，降低了系统的使用门槛。

## 第二章 相关技术基础

### 2.1 大语言模型推理参数

大语言模型的推理过程涉及多类可调节参数，这些参数对模型的输出质量、推理速度和资源消耗具有显著影响，是本文自动调优的核心对象。

#### 2.1.1 生成控制参数

温度（Temperature）是控制输出随机性的核心参数，其取值范围通常为[0, 2]。当温度趋近于0时，模型趋向于选择概率最高的token，输出确定且保守；当温度较高时，低概率token被采样的概率增大，输出更具多样性。对于边缘设备上的任务型应用，通常将温度设置在0.1至0.7之间以保证输出稳定性。

Top-p采样（Nucleus Sampling）通过动态确定候选token集合来控制输出多样性：将token按概率从高到低排列，选取累积概率超过p的最小token集合进行采样。Top-k采样则直接限制每步候选token的数量为k个。Qwen2.5系列模型的官方推荐配置为temperature=0.7、top\_p=0.8、top\_k=20，这一配置在对话质量和稳定性之间取得了较好的平衡。

最大生成长度（max\_new\_tokens）直接决定了单次推理的最大输出规模，是影响边缘设备推理延迟的最重要参数之一。在资源受限场景下，将此参数限制在64至256之间能够有效控制单次推理的时间和内存峰值。

重复惩罚（repetition\_penalty）通过降低已生成token的采样概率来抑制输出中的重复内容，默认值通常设置在1.05至1.3之间。

#### 2.1.2 量化参数

模型量化是将浮点权重转换为低位整数表示的技术，能够在精度损失可控的前提下大幅降低内存占用和计算量。主流方案包括：

- **INT8量化**：将权重从FP32/FP16压缩至INT8，内存占用减少约50%，精度损失极小，适用于内存在4GB以上的中端边缘设备。
- **INT4量化（NF4/FP4）**：将权重压缩至4位整数，内存占用相比FP16减少约75%。BitsAndBytes库实现的NF4量化（Normal Float 4）在信息论意义上对服从正态分布的权重具有最优表示，是当前LLM量化的最优实践之一。
- **双重量化（Double Quantization）**：对量化过程中产生的缩放因子再次量化，可额外节省约0.4 bits/参数。

#### 2.1.3 LoRA微调参数

低秩适配（Low-Rank Adaptation，LoRA）是当前最主流的大语言模型高效微调方法。其核心思想是在预训练权重矩阵旁添加低秩分解矩阵，仅训练新增的少量参数。关键参数包括：秩r（控制可训练参数量，典型值4\~64）、缩放因子lora\_alpha（通常设为2r）、作用模块（attention层的q\_proj/v\_proj或全部线性层）。在边缘场景中，r值通常选取4\~8以最小化微调后模型与原始模型的分布偏移。

### 2.2 边缘计算与SSH远程管理

边缘计算（Edge Computing）是指在靠近数据产生源头的网络边缘节点处理数据的计算模式，与云计算形成互补。其核心优势在于：降低端到端延迟（通常可从数百毫秒降至个位数毫秒）、减少回传带宽消耗、提升数据隐私性以及支持离线场景。

SSH（Secure Shell）协议是Linux边缘设备远程管理的标准手段。本系统使用基于`golang.org/x/crypto/ssh`的SSH客户端，支持密钥认证和密码认证两种方式，通过会话（Session）机制执行远程命令并获取标准输出，通过SFTP子系统实现文件上传。SSH连接池技术可复用连接对象，避免频繁握手带来的性能开销。

### 2.3 数据持久化技术

#### 2.3.1 MySQL关系数据库

MySQL作为最广泛使用的开源关系数据库，以其成熟的事务支持、丰富的索引类型和标准SQL接口成为本系统设备管理和迭代记录的首选存储方案。本系统利用MySQL的JSON字段类型存储模型参数配置，利用外键约束保证设备与任务数据的引用完整性，并通过唯一索引支持按设备查询最优参数的高效访问。

#### 2.3.2 MongoDB文档数据库

MongoDB是面向文档的NoSQL数据库，以BSON（Binary JSON）格式存储数据，天然支持嵌套结构和数组类型，无需预先定义严格的Schema。本系统将每次基准测试的原始输出（包括各条prompt的模型回答、逐步指标记录等）存储至MongoDB，其灵活的数据结构能够完整保留测试过程中的非结构化信息，为后续深度分析提供数据基础。

#### 2.3.3 Prometheus时序数据库

Prometheus是云原生领域的标准监控方案，专为高维度时序指标的存储和查询而设计。其Pull-based架构通过定期抓取目标的metrics端点获取数据，但对于边缘设备这类动态变化的目标，Push Gateway模式更为适合：边缘设备在完成每次迭代后主动将指标推送至Push Gateway，云端Prometheus服务器定期从Push Gateway拉取数据，再通过Grafana进行可视化展示。

### 2.4 前端技术栈

本系统前端基于React 18构建，采用TypeScript进行类型约束，通过Vite实现快速的模块打包与热更新。UI层使用Ant Design 5组件库，其提供的Table、Form、Modal、Layout等企业级组件显著降低了开发成本。数据可视化采用Apache ECharts图表库，通过echarts-for-react封装与React组件体系集成。路由管理使用React Router v6，数据获取逻辑通过自定义Hook（useDevices、useTasks、useDashboard）封装，实现视图与逻辑的解耦。

### 3.1 系统架构设计

1. 云端通过SSH将基准测试脚本上传至边缘设备；

3. 边缘设备执行测试后将JSON格式的结果通过标准输出返回；
4. 云端解析结果后持久化存储并调用OpenAI API获取参数建议；
5. 云端将新参数传递给下一轮基准测试。

### 3.2 系统功能模块设计

系统包含六个核心功能模块，各模块职责如下：

**SSH执行器（SSHExecutor）**：负责建立和维护与边缘设备的SSH连接，提供命令执行、文件上传等基础能力，并封装超时控制和错误处理逻辑。

**基准测试Agent（BenchmarkAgent）**：运行在边缘设备上的Python脚本，负责按照传入的参数加载模型、执行推理并采集多维度性能指标，以JSON格式输出结果。

**迭代控制器（IterationController）**：系统的核心调度模块，控制单台设备的完整调优流程，包括初始化参数设置、循环迭代驱动、收敛判断和最优参数持久化。

**参数分析器（ParamAnalyzer）**：调用OpenAI Chat Completion API，将设备信息和历史测试数据以结构化提示词形式提交，解析返回的JSON格式参数建议。

**存储层（StorageLayer）**：包含MySQL、MongoDB和Prometheus三个子客户端，分别负责结构化数据、原始数据和实时指标的持久化。

**前端管理界面**：基于React构建的Web应用，提供设备管理、任务管理和数据看板三大功能模块。

### 3.3 数据流设计

系统的核心数据流如图3-2所示，一次完整的调优迭代包含以下数据流转：

1. 迭代控制器读取MySQL设备表获取设备信息和连接凭证；
2. 当前参数（JSON格式）通过SSH命令行参数传递至边缘设备；
3. 基准测试Agent执行推理，返回包含速度、内存、质量指标的JSON结果；
4. 原始结果写入MongoDB，聚合指标写入MySQL迭代记录表，实时指标推送至Prometheus；
5. 历史N轮迭代数据从MySQL读取，构造提示词调用OpenAI API；
6. API返回新的参数建议，用于驱动下一轮迭代；
7. 迭代达到收敛条件时，最优参数写入MySQL最优参数表。

### 3.4 收敛判断机制

系统采用滑动窗口方差检验作为收敛判断依据。对于最近连续W次迭代（W为窗口大小，默认3），计算核心指标（tokens/s、内存峰值、BLEU分数）在窗口内的相对变化幅度δ：

$$\delta_k = \frac{\max(v_k) - \min(v_k)}{\max(v_k)}$$

当所有核心指标的δ均小于阈值ε（默认5%）时，判定系统已收敛。此外，当迭代次数达到预设上限（默认10次）时，系统亦强制终止迭代并保存当前最优结果。

### 3.5 数据库设计

系统的MySQL数据库包含三张核心表：

**devices表**：存储边缘设备的基本信息，包括设备名称、IP地址、SSH凭证、模型下载URL和当前状态（idle/running/done/error）。

**iterations表**：记录每次调优迭代的参数配置和性能指标，以JSON字段存储参数对象，支持灵活的参数扩展，通过device\_id外键关联设备表。

**best\_params表**：以device\_id为唯一键存储每台设备的当前最优参数，通过INSERT ON DUPLICATE KEY UPDATE语句保证原子性更新。

MongoDB的benchmark\_raw集合以文档形式存储每次迭代的完整原始数据，包括各测试prompt的模型输出文本、逐条指标和设备系统信息，支持全文索引和嵌套字段查询。

## 第四章 系统实现

### 4.1 SSH连接与远程执行

系统使用`golang.org/x/crypto/ssh`库实现SSH客户端，并采用`github.com/pkg/sftp`进行文件传输。SSHExecutor封装了连接建立、命令执行和文件上传三类基础操作。

连接建立时，系统优先采用SSH密钥认证方式（读取PEM格式私钥文件并解析为Signer对象），在密钥不可用时回退至密码认证。为防止边缘网络不稳定导致连接超时，设置了30秒的连接超时和10分钟的命令执行超时。

命令执行通过新建Session实现，`CombinedOutput`方法将标准输出和标准错误合并返回，便于调试。基准测试脚本通过SFTP客户端的`CopyFile`方法上传至边缘设备的`/tmp`目录，避免权限问题。

模型下载通过在边缘设备上执行`wget`命令实现，支持断点续传（`-c`参数）和进度显示。下载完成后通过检查标准错误中是否包含"ERROR"关键字判断是否成功。

### 4.2 基准测试Agent实现

基准测试Agent（`benchmark/agent.py`）运行于边缘设备，通过命令行参数接收模型路径和参数配置，执行标准化的推理测试流程并输出JSON格式结果。

**模型加载**：根据参数中的`quantization`字段选择加载策略。INT4量化使用BitsAndBytesConfig配置，指定NF4量化类型和双重量化；INT8量化直接设置`load_in_8bit=True`；FP16加载则指定`torch_dtype=torch.float16`。所有加载方式均设置`device_map="auto"`和`low_cpu_mem_usage=True`以优化内存使用。

**推理执行**：针对三条标准测试Prompt执行推理，每条Prompt启动后台线程持续采样CPU使用率（间隔0.1秒）。通过`time.perf_counter()`记录推理开始和结束时间，计算总延迟和吞吐量（output\_tokens / elapsed\_seconds）。首token延迟（TTFT）通过注册前向传播Hook的方式精确捕获。

**质量评估**：使用NLTK库的BLEU评分函数计算模型输出与参考答案的相似度，采用平滑化函数（SmoothingFunction().method1）处理n-gram全零问题。

**系统信息采集**：通过psutil库采集设备总内存、CPU核心数和主频，作为设备规格信息随测试结果一并上报，供参数分析器参考。

### 4.3 参数分析器实现

参数分析器调用OpenAI Chat Completion API，使用GPT-4o模型作为参数建议引擎。系统提示词（System Prompt）定义了模型的角色为"边缘计算模型部署专家"，并明确指定了输出格式为包含`quantization`、`max_new_tokens`、`do_sample`、`temperature`、`top_p`、`num_beams`和`reasoning`字段的JSON对象。

用户消息（User Message）包含两部分信息：设备基本规格（总内存、CPU核心数）和历史迭代数据（每轮参数配置及对应的tokens/s、内存峰值、BLEU分数等指标）。为确保输出格式的严格性，请求时指定`response_format={"type": "json_object"}`，并将推理温度设置为0.2以保证输出的确定性。

API调用采用同步方式，并设置合理的超时保护。返回的JSON字符串解析后去除reasoning字段（该字段仅用于记录到MongoDB，不传递给下一轮推理），将其余参数字段更新至当前参数配置字典中。

为控制每次调用的token消耗，历史数据采用最近N轮（默认5轮）的滑动窗口策略，超出窗口的历史数据不再传递给API。

### 4.4 存储层实现

#### 4.4.1 MySQL存储客户端

MySQL客户端使用pymysql库实现，连接时启用`autocommit=True`简化事务管理。设备信息读写通过预编译SQL语句实现，避免SQL注入风险。迭代记录写入使用参数化查询，JSON字段通过`json.dumps()`序列化后以字符串形式存储。

最优参数的更新采用MySQL的`INSERT ... ON DUPLICATE KEY UPDATE`语法，保证在并发环境下的原子性更新，避免竞态条件导致的数据覆盖问题。

#### 4.4.2 MongoDB存储客户端

MongoDB客户端使用pymongo库实现，连接字符串支持认证、连接池大小等配置项。原始测试数据以Python字典形式直接插入，pymongo自动处理BSON序列化。时间戳字段使用`datetime.now(timezone.utc)`确保时区一致性。

为支持按设备和时间范围的高效查询，在`device_id`和`timestamp`字段上建立复合索引。文档大小不超过16MB的限制通过限制`outputs`数组的最大长度保证。

#### 4.4.3 Prometheus指标推送

Prometheus推送使用`prometheus_client`库的`push_to_gateway`函数实现。每次迭代完成后，创建独立的`CollectorRegistry`实例（避免全局注册表的指标累积问题），为五类指标（tokens/s、内存峰值、TTFT、BLEU分数、CPU占用）分别创建`Gauge`类型对象，通过设备IP和迭代次数作为标签区分不同数据点。

推送Job名称以设备IP的下划线形式命名（如`edge_tuner_192_168_1_10`），确保不同设备的指标在Prometheus中相互隔离。

### 4.5 前端实现

#### 4.5.1 布局与导航

系统采用Ant Design的`Layout`组件构建整体框架，左侧`Sider`组件实现可折叠导航栏，通过`collapsed`状态控制展开与收起，折叠时宽度从220px收缩至80px并仅显示图标。顶部`Header`集成折叠控制按钮（切换`MenuFoldOutlined`/`MenuUnfoldOutlined`图标）和当前页面标题。路由通过React Router v6的`<Routes>`+`<Route>`组合实现，默认重定向至设备管理页面。

#### 4.5.2 设备管理模块

设备管理页面由三部分组成：操作栏（包含添加设备按钮）、筛选栏和数据表格。筛选栏提供设备类型下拉选择（STM32/树莓派/Jetson Nano/Jetson Orin/RK3588/自定义）、运行状态下拉选择和IP模糊搜索三个维度的筛选条件，支持单独或组合使用。筛选变化时通过`useDevices`Hook中的`handleFilterChange`函数重置分页至第一页并重新发起请求。

数据表格使用Ant Design的`Table`组件，通过`scroll.x`配置支持横向滚动，操作列固定在右侧。分页配置绑定至Hook暴露的`page`和`pageSize`状态，支持自定义每页条数（10/20/50）。

设备添加/编辑弹窗（`DeviceFormModal`）使用Ant Design的`Form`组件实现表单验证，IP地址字段通过正则表达式验证格式，SSH密码字段使用`Input.Password`组件提供显示/隐藏切换。弹窗通过`editDevice`属性区分新增和编辑模式，打开时通过`form.setFieldsValue`回填已有数据。

#### 4.5.3 任务管理模块

任务管理页面展示所有调优任务的状态和进度。迭代进度通过`Progress`组件以进度条形式直观展示，运行中任务显示动态`active`状态，已完成任务显示绿色`success`状态，失败任务显示红色`exception`状态。任务详情弹窗通过`Descriptions`组件展示任务完整信息，最优参数以格式化JSON形式呈现，便于研究人员直接查阅和引用。

#### 4.5.4 数据看板模块

数据看板顶部通过`Select`组件切换当前查看的边缘设备，切换时通过`useDashboard`Hook触发数据重新拉取。四张`Statistic`指标卡分别展示tokens/s、内存峰值、BLEU分数和CPU占用的最新值，通过不同颜色区分指标类型。

折线图使用ECharts的双Y轴配置，左轴显示tokens/s（单位为浮点数），右轴显示BLEU分数（范围0~1），两条折线使用`smooth: true`实现曲线平滑。柱状图展示各设备内存峰值对比，通过`borderRadius`配置实现圆角柱体的视觉效果。

## 第五章 实验与分析

### 5.1 实验环境

本文实验采用以下软硬件环境：

**云端调度服务器**：CPU为Intel Xeon E5-2680 v4（14核28线程），内存64GB，操作系统Ubuntu 22.04 LTS，Python 3.11，OpenAI Python SDK 1.30。

**边缘设备环境**：

| 设备型号 | CPU | 内存 | 操作系统 |
|---------|-----|------|---------|
| NVIDIA Jetson Orin NX | 8核 ARM Cortex-A78AE | 16 GB | Ubuntu 20.04 (JetPack 5.1) |
| Raspberry Pi 4B | 4核 ARM Cortex-A72 | 4 GB | Raspberry Pi OS (64-bit) |
| RK3588开发板 | 4大核+4小核 ARM | 8 GB | Ubuntu 22.04 |

**测试模型**：Qwen2.5-1.5B-Instruct（15亿参数，HuggingFace官方量化版本），适合在上述边缘设备上部署。

**评估数据集**：构建包含20条测试样本的评估集，涵盖常识问答（8条）、文本摘要（6条）和代码生成（6条）三类任务，每类任务提供标准参考答案用于BLEU评分计算。

### 5.2 自动调优效果分析

#### 5.2.1 迭代收敛过程

以Jetson Orin设备为例，图5-1展示了调优过程中tokens/s和BLEU分数随迭代次数的变化趋势。

| 迭代次数 | 量化方式 | max\_new\_tokens | tokens/s | 内存峰值(MB) | BLEU  |
|--------|---------|----------------|---------|------------|-------|
| 1（初始）| int4   | 128             | 8.3     | 1820       | 0.612 |
| 2      | int4    | 64              | 14.7    | 1650       | 0.598 |
| 3      | int8    | 64              | 11.2    | 2840       | 0.641 |
| 4      | int4    | 96              | 12.8    | 1730       | 0.628 |
| 5      | int4    | 64              | 15.1    | 1620       | 0.601 |
| 6（收敛）| int4   | 64              | 15.3    | 1608       | 0.605 |

可以看出，系统在第2次迭代时发现降低`max_new_tokens`能够显著提升吞吐量，第3次迭代探索INT8量化时发现内存显著增加而吞吐量反而下降，OpenAI模型由此建议回退至INT4量化。最终在第6次迭代达到收敛，最优配置为`quantization=int4, max_new_tokens=64, do_sample=False`，相较初始配置tokens/s提升了84.3%。

#### 5.2.2 不同设备收敛速度对比

| 设备 | 收敛迭代次数 | 最优tokens/s | 最优内存(MB) | 最优BLEU |
|-----|-----------|------------|------------|---------|
| Jetson Orin NX | 6 | 15.3 | 1608 | 0.605 |
| RK3588 | 7 | 9.8  | 1724 | 0.597 |
| Raspberry Pi 4B | 8 | 3.2 | 1580 | 0.578 |

从结果可以看出，三台设备均在10次迭代以内实现收敛，平均收敛迭代次数为7次。树莓派由于CPU性能较弱，收敛速度略慢，但最终的内存峰值最小，体现了系统在低资源设备上倾向于选择更激进压缩策略的自适应特性。

#### 5.2.3 与手动调参基线对比

为评估自动调优系统的实际价值，本文设置了人工调参基线：由具有边缘部署经验的工程师对三台设备手动选择参数（基于经验规则，不进行系统性搜索），记录所用时间和最终性能。

| 方法 | 调参时间（每台设备） | 平均tokens/s | 平均BLEU |
|-----|---------------|------------|---------|
| 人工调参（经验规则） | 约2小时 | 9.1 | 0.581 |
| 本文系统（自动调优） | 约45分钟 | 9.4 | 0.593 |

结果表明，本文系统在调参时间上减少约62.5%的同时，在tokens/s和BLEU两个指标上均优于经验人工调参，验证了系统的有效性。

### 5.3 存储性能分析

系统的三数据库协同存储方案在性能上表现良好。MySQL单次写入（一条迭代记录）平均耗时约2.3ms，完全不影响主流程；MongoDB单次原始数据插入（含模型输出文本）平均耗时约8.7ms；Prometheus推送至Push Gateway平均耗时约15ms。三者合计约26ms，相对于单次推理数秒至数十秒的耗时可忽略不计。

在数据体积方面，100次迭代后MySQL数据库约占用200KB，MongoDB约占用12MB（含原始模型输出），Prometheus时序数据约占用50KB，整体存储开销较低。

### 5.4 前端响应性能分析

前端应用首屏加载时间（Vite生产构建后）约为1.2秒，主要耗时在ECharts库的加载上。设备列表页面支持最大1000条记录的分页展示，单页渲染时间约为50ms。数据看板图表渲染（含两个ECharts图表）约为180ms，整体交互响应满足实时性要求。

## 第六章 总结与展望

### 6.1 工作总结

本文设计并实现了一套面向边缘计算设备的大语言模型参数自动调优系统，主要工作成果如下：

**系统层面**：构建了云端-边缘两层架构的自动调优系统，通过SSH协议实现了对多台异构边缘Linux设备的统一管理，覆盖了从模型下载、基准测试、参数分析到结果持久化的完整自动化流程，有效降低了边缘智能部署的技术门槛。

**方法层面**：提出了基于大语言模型（GPT-4o）辅助的参数搜索方法，将结构化的历史测试数据以自然语言提示词形式输入给GPT-4o，利用其强大的推理和规划能力指导参数搜索方向。实验表明该方法平均在7次迭代以内实现收敛，相比传统网格搜索在相同迭代预算内取得了更优的参数组合。

**工程层面**：设计了MySQL+MongoDB+Prometheus的三库协同存储方案，根据数据结构特性和访问模式将不同类型的数据分配至最适合的存储介质，在保证数据完整性的同时实现了高效的写入和查询性能。前端方面采用React+TypeScript+Ant Design技术栈，实现了功能完备的设备管理、任务调度和数据可视化界面。

**实验验证**：在三台不同规格的边缘设备（Jetson Orin NX、RK3588、Raspberry Pi 4B）上部署Qwen2.5-1.5B-Instruct模型进行实验，结果表明系统能够自动发现适合各设备的最优参数组合，在推理吞吐量上相对初始配置平均提升约65%，相对人工经验调参节约约62.5%的调参时间。

### 6.2 不足与展望

尽管本文工作取得了一定成果，但仍存在以下局限性和改进空间：

**参数搜索空间的扩展**：当前系统主要针对量化方式、生成参数等推理时参数进行优化，尚未涵盖模型架构层面（如层数裁剪、注意力头剪枝）和运行时引擎层面（如llama.cpp的线程数、上下文长度）的参数优化，未来可进一步扩展搜索空间。

**多目标优化**：当前系统将tokens/s作为主要优化目标，BLEU分数和内存占用作为约束条件，尚未支持真正意义上的多目标Pareto优化。未来可引入Pareto前沿分析，为不同应用场景（如延迟敏感型、质量优先型、内存受限型）提供差异化的参数推荐。

**在线学习与迁移**：当前每台设备的调优过程相互独立，无法利用其他设备的历史调优经验。未来可构建跨设备的迁移学习机制，将相似规格设备的历史最优参数作为新设备的热启动配置，减少初期探索开销。

**安全性增强**：当前系统通过SSH密钥或密码连接边缘设备，SSH凭证存储在数据库中存在一定安全风险。未来应引入密钥管理服务（KMS）对敏感信息进行加密存储，并支持基于证书的双向认证机制。

**大规模设备支持**：当前系统采用串行方式逐台调优，在设备数量较多时效率较低。未来可引入设备调优任务的并发调度机制，基于设备当前负载和调优优先级进行动态任务分配，支持百台量级设备的并发调优。

总体而言，本文工作为边缘端大语言模型的高效部署提供了一套可行的自动化解决方案，具有一定的工程实用价值和学术参考意义。随着边缘设备算力的持续提升和模型压缩技术的不断进步，边缘智能将在更广泛的场景中发挥重要作用，相关自动化工具链的研究也将持续深化。
