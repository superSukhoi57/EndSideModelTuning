在 Kubernetes (K8s) 中，Pod 的状态（Phase）反映了它当前所处的生命周期阶段。理解这些状态对于排查问题至关重要。

通常我们将状态分为 **5 种核心状态** 和 **常见的具体原因状态**。

### 📊 5 种核心状态

这是 K8s 定义的 Pod 生命周期宏观阶段：

| 状态                 | 含义                                   | 典型场景                           |
| -------------------- | -------------------------------------- | ---------------------------------- |
| **Pending (等待)**   | Pod 已被 K8s 接收，但尚未完全运行。    | 正在调度、正在下载镜像、资源不足。 |
| **Running (运行中)** | Pod 已绑定节点，容器已创建。           | 至少有一个容器正在运行或重启中。   |
| **Succeeded (成功)** | 所有容器都成功终止，且不再重启。       | 一次性任务（Job）执行完毕。        |
| **Failed (失败)**    | 所有容器都终止，至少有一个因错误退出。 | 程序崩溃、配置错误、被系统杀死。   |
| **Unknown (未知)**   | 无法获取 Pod 状态。                    | 节点断网、Kubelet 进程挂掉。       |

------

### 🔍 常见具体状态与原因

在实际运维中，你更多看到的是 `Pending` 或 `Running` 下的具体细分状态（Reason），这些通常需要通过 `kubectl describe pod <pod-name>` 命令查看。

#### ⏳ 处于 Pending 阶段时

- **ContainerCreating**
  - **含义**：正在创建容器。
  - **常见原因**：正在拉取镜像（网络慢）、挂载存储卷（PVC 绑定中）。
- **ImagePullBackOff**
  - **含义**：拉取镜像失败，系统正在重试。
  - **常见原因**：镜像名称写错、镜像不存在、私有仓库认证失败。
- **ErrImagePull**
  - **含义**：拉取镜像遇到错误。
  - **常见原因**：同上，通常是网络问题或镜像源不可达。

#### 🔄 处于 Running 阶段时（看似运行，实则异常）

- **CrashLoopBackOff**
  - **含义**：容器启动后立刻崩溃，K8s 正在按指数退避策略（越来越慢）尝试重启它。
  - **常见原因**：应用程序报错退出、配置文件缺失、健康检查失败。
- **OOMKilled**
  - **含义**：内存溢出被杀。
  - **常见原因**：程序内存泄漏，或设置的内存限制（Limit）太小。
- **Error**
  - **含义**：容器启动过程中发生错误。
  - **常见原因**：启动命令写错、权限不足。
- **Completed**
  - **含义**：通常出现在 `Running` 状态列，表示任务已成功结束。
  - **常见原因**：Job 或 CronJob 任务跑完了。

#### 🛑 终止与驱逐

- **Terminating**
  - **含义**：Pod 正在被删除。
  - **过程**：发送停止信号 -> 等待优雅退出（默认 30 秒） -> 强制杀死。
- **Evicted (被驱逐)**
  - **含义**：Pod 被节点“踢”出来了。
  - **常见原因**：节点资源不足（磁盘满了、内存不够），系统为了保命杀掉了低优先级的 Pod。

### 🛠️ 快速排查指南

当你看到 Pod 状态不对劲时，不要慌，按以下顺序操作：

1. **看详情**：

   ```bash
   kubectl describe pod <pod-name> -n <namespace>
   ```

   *重点看底部的 `Events` 和 `Reason` 字段，它会告诉你为什么是 Pending 或 Failed。*

2. **看日志**：

   ```bash
   kubectl logs <pod-name> -n <namespace>
   ```

   *如果是 `CrashLoopBackOff`，建议加上 `--previous` 查看崩溃前的日志：*

   ```bash
   kubectl logs <pod-name> -n <namespace> --previous
   ```

3. **看节点**：
   如果状态是 `Unknown`，检查一下节点是不是挂了：

   ```bash
   kubectl get nodes
   ```
