### ：AccessToken + RefreshToken 双 Token（**目前企业主流标准**）

#### 标准设计

1. **AccessToken（访问令牌）**

   

   - 时效极短：`15分钟 ~ 1小时`
   - 用于**接口鉴权**，每次请求携带
   - 短期有效，就算泄露，危害窗口很小

   

2. **RefreshToken（刷新令牌）**

   

   - 时效很长：`7天 ~ 30天`
   - **不参与接口鉴权**，只用来换新的 AccessToken
   - 一般存在 `HttpOnly Cookie`，前端 JS 无法读取，防 XSS




#### 完整流程

1. 账号密码登录
2. 后端返回：
   - `accessToken`（短时效，存 localStorage）
   - `refreshToken`（长时效，存 HttpOnly Cookie）
3. 正常业务请求：带 accessToken
4. accessToken 过期 → 前端自动调用「刷新接口」
5. 用 refreshToken 换取**新 accessToken**，无感续期
6. 退出登录：双 token 一起销毁、后端拉黑刷新令牌

#### 核心优势

- **安全性拉满**：短期 access 泄露影响极小
- 用户无感知，不会突然掉线
- 支持主动踢人、下线设备
- 大厂 / 开源框架统一规范（SpringSecurity、Sa-Token、Node 生态全标配）



方案A：返回当前的 refresh token（不重新生成）

- 优点：简单高效，无需额外计算
- 缺点：token 的 iat （签发时间）不变

方案B：重新生成新的 refresh token（当前实现）

- 优点：Token Rotation（令牌轮换），每次刷新都产生新 token，旧 token 失效，安全性更高，可防止 token 泄露后被重复使用
- 缺点：多一次签名计算
  推荐方案B （当前实现），这是业界最佳实践。OAuth 2.0 规范也推荐 refresh token rotation，可以降低 token 被盗用的风险。

#### 退出登录设计

将token的key设计为  accesstoken：userid和refreshtoken：userid，前端就可以直接通过accesstoken就可以直接退出登录，不需要穿refreshtoken！！！！

## 踩坑	

**主模块里面有子模块的引用，需要一起删掉才能被生效**

`frontend` 目录之前是一个 **git 子模块（submodule）**。然后我删除了frontend里面的.git还是不能被外部的git管理，执行git add frontend没有文件被提交。

**子模块是什么：**

- 子模块是一个独立的 git 仓库，被嵌入到另一个主仓库中
- `frontend` 目录有自己的 `.git` 目录、自己的提交历史和自己的远程仓库
- 在主仓库中，子模块只被记录为一个**特殊的引用指针**（指向子模块的某个 commit）

**为什么不能提交子模块里的文件：**

- 当你修改 `frontend/src/App.js` 时，这个修改属于 `frontend` 子模块仓库，不属于主仓库
- 主仓库只记录 `frontend` 指向哪个 commit，不记录里面的具体文件
- 所以你在主仓库执行 `git add frontend/src/App.js` 会报错，因为 git 认为这些文件属于另一个仓库

**我刚才做了什么：**

1. 用 `git rm --cached frontend` 从主仓库的索引中删除了子模块引用
2. 用 `git add frontend/` 把整个目录作为普通文件夹重新添加到主仓库

现在 `frontend` 就是一个普通的目录了，里面的所有文件都可以正常提交。
