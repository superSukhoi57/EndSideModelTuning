# 飞书 OAuth2.0 扫码登录 **完整流程 + 前后端分离跳转方案**

我给你用**最简单、最贴合你当前项目**的方式讲清楚：

你现在的场景是：

- **前端**：React 页面（展示飞书二维码）
- **重定向 URI**：配置的是**后端接口**（不是前端）
- **需求**：后端处理完成后，**前端能知道登录成功，并跳到首页**

下面我把**整个流程、为什么要这样、前端怎么拿到状态、怎么跳转**一次性讲透。

### 第 1 步：前端页面生成飞书二维码

```
window.QRLogin({ goto: "飞书授权地址" })
```

作用：生成二维码，等待用户飞书 APP 扫码授权。

------

### 第 2 步：用户扫码 → 飞书授权 → 飞书调用 **你的重定向地址（后端）**

飞书会**自动重定向**到你填的：

```
http://47.115.225.81:12300/callback?code=xxxx&state=xxx
```

**✅ 重点：**

**重定向地址一定是后端接口，不能是前端！**

因为：

- 后端需要用 `code` 换取 **access_token、userid、用户信息**
- 这个过程必须**后端保密**，不能暴露在前端

------

### 第 3 步：后端处理 /callback 接口（核心步骤）

后端要做 4 件事：

1. 接收飞书传回的 `code`
2. 用 code 请求飞书接口 → **获取用户信息**
3. 生成自己业务的 **token / session**
4. **重定向回前端页面**，并把 token 带给前端

------

### 第 4 步：后端重定向回前端（最关键！）

这就是你问的：

### **重定向 URI 是后端，前端怎么跳？**

答案：**后端处理完登录 → 直接重定向到前端页面，并带上 token**

```
前端登录页：http://47.115.225.81:3000/login
前端首页：http://47.115.225.81:3000/
```

后端 `/callback` 最后一行代码：

```
// Go 示例
http.Redirect(w, r, "http://47.115.225.81:3000/?token=Bearer xxx", 302)
```

```
// Java 示例
response.sendRedirect("http://47.115.225.81:3000/?token=Bearer xxx");
```

```
// Node.js 示例
res.redirect("http://47.115.225.81:3000/?token=Bearer xxx");
```

------

### 第 5 步：前端拿到 token → 保存 → 跳转到首页

前端首页（或登录页）监听 URL 参数：

1. 取出 `token`
2. 存入 localStorage
3. 跳转到首页或控制台

------

## 二、最终完整流程（你一看就懂）

```
1. 前端打开 /login → 显示飞书二维码
2. 用户扫码 → 飞书授权 → 重定向 → 后端 /callback
3. 后端 /callback 处理登录 → 生成 token → 重定向 → 前端页面
4. 前端获取 token → 保存 → 跳转到首页
```

------

## 三、前端怎么知道登录成功？

### 方法 A（最标准、最安全）

后端重定向回前端，**URL 带上 token**

```
http://47.115.225.81:3000/?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
```

前端代码：

```
useEffect(() => {
  const params = new URLSearchParams(window.location.search);
  const token = params.get("token");

  if (token) {
    localStorage.setItem("token", token);
    window.location.href = "/home"; // 跳首页
  }
}, []);
```

------

### 方法 B（更优雅）

后端重定向时带 **status=success**

```
http://47.115.225.81:3000/login?status=success&token=xxx
```

前端判断 status 即可。

------

## 四、一句话总结（最重要）

## **重定向 URI 是后端 → 后端处理完 → 重定向回前端 → 前端跳转首页**

这就是 **前后端分离 + OAuth2.0 登录** 的标准方案！

# 整体流程是

1、一个state （可以后端生成给前端或者前端生成给后端），最终后端也会有这个state

2、使用这个state初始化sdk，扫描后回调到后端的（前端sdk的回调地址是接收code和state的！）

3、扫码后出发回调，重定向请求后端的接口

4、后端拿state和请求的对比，判断是否终止

5、通过则生成token然后带在响应里面让前端重定向（这个是前端的地址）到前端

同一个人，在**同一个企业**的所有应用里 union_id **完全一样**。







## 设计

设计类似层级结构时，不变的一定要放在前面

像redis的key auth:state:XXXXXXX

之前kafka/mqtt的主题：一级主题/${sn}/功能主题/其他主题。这样设计不好，不变的放在后面，后面在使用事件总线和flink处理时，对于第二个需要变的很不好处理





打开当前登录页面就请求一个state，然后在后端redis只有5min，二维码在这期间过期了没事，5min内三百年刷新。但是在网页上如果停留超过5min就不能登录，需要重新刷新页面。
